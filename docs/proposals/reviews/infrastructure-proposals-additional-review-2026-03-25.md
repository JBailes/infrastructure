## New high-value issues still open

### 1) `spire-server` VM boot order likely causes installer re-entry loops

`qm create` sets boot order to `ide2;scsi0` (CD-ROM first). After installation, unattended reboot can return to installer media instead of booting disk unless the operator manually changes order/ejects ISO.

- **Where:** `proxmox-deployment-automation.md` VM creation snippet.
- **Why it matters:** avoidable deployment interruption and confusing recovery state.
- **Fix:** set initial install order to CD-ROM, then script/post-step flips permanently to `scsi0` and detaches ISO.

### 2) "Script integrity" check is not tamper-evident against same-host compromise

The orchestrator validates script checksums against `checksums.sha256`, but both scripts and checksum manifest live in the same mutable tree and are regenerated by operator command.

- **Where:** integrity section in Proxmox proposal.
- **Why it matters:** an attacker modifying `BOOTSTRAP_DIR` can update both script and manifest, passing validation.
- **Fix:** sign the manifest with an offline key (minisign/cosign/GPG) or pin verified commit hashes.

### 3) Secret scrub guarantees are incomplete for unreachable/offline hosts

The design relies on trap + post-run scrub (`pct exec`/SSH) for `/root/.env.bootstrap`, but does not define success criteria when a target is down or unreachable during scrub.

- **Where:** post-deploy secret scrub section.
- **Why it matters:** secrets can persist indefinitely on failed hosts, then be exposed on later forensic access or disk theft.
- **Fix:** require scrub status reporting per host, block "success" state on unsanitized hosts, and enforce boot-time self-destruct check for stale bootstrap env files.

### 4) Root execution model has no constrained command channel

The deployment model uses root remote execution for bootstrap and notes this as intentional. No forced-command key, command allowlist, or per-phase ephemeral SSH cert/key is defined.

- **Where:** root execution trade-off section.
- **Why it matters:** compromise of orchestrator-side root key materially expands blast radius and persistence options.
- **Fix:** short-lived SSH certificates + forced command wrappers; rotate deployment credentials per run.

### 5) World bulk snapshot sessions can become a connection-pool DoS lever

The world API holds a repeatable-read transaction open across multiple `/bulk/*` calls with a 30s idle timeout.

- **Where:** world bulk consistency design.
- **Why it matters:** a small number of malicious or buggy clients can open/idle snapshot sessions and pin DB connections/transaction state, degrading startup for legitimate realms.
- **Fix:** hard cap concurrent snapshot sessions, per-caller quota, tight absolute TTL, and explicit cancellation endpoint.

### 6) Snapshot-token affinity requirement undermines HA/load-balancing posture

The design requires all bulk calls for a startup session to hit the same API instance, and explicitly tells operators to avoid load-balanced VIP unless special affinity is added.

- **Where:** world HA consideration for snapshot token.
- **Why it matters:** this couples startup correctness to routing stickiness and makes horizontal scaling/error recovery fragile.
- **Fix:** externalize snapshot state (Redis/DB-backed session mapping) or provide a single "bulk world snapshot" endpoint to eliminate multi-call affinity.

### 7) Rate limiting semantics are not deployment-safe across multiple API instances

Accounts and players proposals define per-caller/per-endpoint limits, but do not specify a shared limiter store or synchronization model.

- **Where:** accounts/players rate-limiting sections.
- **Why it matters:** with N replicas, effective attacker budget scales roughly by N if counters are process-local.
- **Fix:** use distributed counters (Redis/sliding-window) and define fail-open/fail-closed behavior on limiter backend outages.

### 8) Proxmox automation is intentionally non-convergent, but no drift-audit control is mandated

Create-once semantics explicitly skip existing hosts and do not reconcile changed inventory values.

- **Where:** proxmox goals and create-once note.
- **Why it matters:** inventory can silently diverge from runtime (CPU/RAM/NIC/disk/security flags), causing "documented" but incorrect state.
- **Fix:** add a required `pve-audit-hosts.sh` drift report that compares live config vs inventory and fails CI/release gates on mismatch.

### 9) Manual VM installation path lacks reproducibility contract

The proposal accepts manual Debian install for the one VM and validates assumptions only at first bootstrap run.

- **Where:** proxmox trade-offs section.
- **Why it matters:** two operators can produce materially different base images (package set, SSH hardening, partitioning), leading to non-repeatable behavior and incident recovery variance.
- **Fix:** publish a minimal deterministic install checklist with machine-verifiable post-install baseline (package/version fingerprint + config hash).

### 10) Log-redaction strategy is pattern-based and can miss structured secret variants

The orchestrator redacts known patterns (long base64, `password=`, `token=`), but there is no guarantee for JSON fields, multiline values, uncommon key names, or encoded fragments.

- **Where:** Proxmox log redaction section.
- **Why it matters:** false negatives in redaction are common; one missed secret in central logs becomes long-lived compromise material.
- **Fix:** invert model: denylist is secondary; primary control should be "no secret output" plus explicit structured logger with field-level redaction before emission.

---

## Suggested order of operations

1. Fix boot-order/script-integrity/scrub semantics in Proxmox proposal first (low effort, high risk reduction).
2. Harden world bulk snapshot protocol against session abuse and affinity fragility.
3. Define distributed rate-limiter contract for multi-instance APIs.
4. Add mandatory drift-audit + reproducibility baseline for deployment consistency.
